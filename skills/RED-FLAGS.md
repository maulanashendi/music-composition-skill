# Red flags — common excuses vs. reality

Shared across the 3 MDLC skills — `jazz-composing`, `plan-verifying`,
`rendering-audition` — plus the legacy `abc-notation` path. When you catch
yourself thinking one of the excuses below, stop and check the reality
column before delivering.

| Excuse (what you might tell yourself) | Reality (what's actually true) | Where to check |
|---|---|---|
| "This section needs every device to prove it's sophisticated." | Stacking every technique is how a section turns into noise. The arc decides what's used; most sections need 2-3 moves, not all of them. | skills/jazz-composing/SKILL.md "the one rule"; skills/jazz-composing/references/arrangement.md interaction-map rule |
| "The chorus should basically repeat the verse — it's a loop, that's fine." | A dramatic arc that never changes isn't an arc. Verse and chorus (or any repeated section) need audible development — register, density, harmony, or interaction shift — not identical restatement. | skills/jazz-composing/references/loop-development.md, arrangement.md |
| "The drum grid and the ABC don't line up bar-for-bar, but it's close enough." (legacy ABC path) | Close enough drifts into audible desync as the piece goes on. Bar counts between the drum step-grid and the ABC must match exactly, section by section. On the current default path this concern moves to `plan.json`/`pyengine`, which enforces bar-matching structurally. | skills/abc-notation/references/drums-and-abc.md, common-failures.md (legacy ABC path only) |
| "Chord symbols in the ABC will just be read as chords, not played as notes." (legacy ABC path) | Some converters (including this package's own `abc_to_midi.py`) will sound a bare chord-symbol string as literal notes unless it's stripped. Always verify the converter's actual output, not just the ABC's visual intent. | skills/abc-notation/scripts/abc_to_midi.py (chord-symbol stripping) |
| "`validate_abc.py` passed with 0 errors, so the piece is fine." (legacy ABC path) | The validator checks ABC structure only — it does not know whether `abc_to_midi.py` will actually produce notes for every voice, or whether the merged MIDI's tempo/time-signature tag matches the ABC header. A file can pass validation and still render silently wrong. Always inspect the actual MIDI output (track count, note count per track, tempo/time-signature) as a separate check. On the plan.json default path, the equivalent trap is "`pyengine validate` passed, so the render is fine" — validate only checks structural/craft warnings, not whether the actual rendered audio sounds right; ear-check is still required. | tests/results/2026-07-13-brief-01-02-armA-vs-armB.md — this exact failure was found twice by independent eval runs before being fixed |
| "The multi-voice ABC looks right — voices are declared, that's enough." (legacy ABC path) | This package's `abc_to_midi.py` only captures lines that literally start with `[V:id]`; a continuation line without the marker is silently dropped (0 notes for that voice) even though the file still validates. Repeat `[V:id]` on every content line for that voice. | skills/abc-notation/references/abc-syntax.md §14, common-failures.md |
| "The rubric score is high, so the piece must sound good." | The rubric checklist scores craft and theory-consistency from the plan/notation — it cannot hear timbre, groove feel, or mix. A high score is a floor (nothing is broken or thin), not a ceiling (nobody has confirmed it's enjoyable). Ear-check is still required before calling a piece finished. | skills/rendering-audition/references/rubric-checklist.md; skills/rendering-audition/references/scorecard-template.md; skills/rendering-audition/references/audition-protocol.md |
| "Motif ini 4-not, arpeggio-nya outside, chord-nya borrowed — sesuai yang kutulis." | Label teori di prosa sering tak cocok notasi aktual: motif "4-not" yang nyatanya 3 pitch (rest bukan pitch), arpeggio chord-tone yang dilabeli "outside", chord diatonik yang disebut "borrowed", klaim top-note yang beda dari voicing tertulis. Jangan percaya label sebelum dicek terhadap fakta notasi yang sebenarnya berbunyi. | skills/abc-notation/scripts/notation_facts.py (jalankan pada ABC-nya; gate wajib di fase Ideation — legacy ABC path. Di jalur default `plan.json`, kesetaraan gate ini adalah `pyengine validate` + audit rubrik/telinga di `rendering-audition`) |
| "First draft-nya sudah solid — tidak ada yang perlu direvisi." | Run tanpa satu pun revisi tercatat berarti temuan rubrik/blind/cliche tidak pernah benar-benar dijawab — "first draft dianggap final" adalah pola gagal, bukan bukti kualitas. Sebelum run disebut selesai wajib ada bagian "Bukti revisi": ≥1 pasangan before/after untuk 2 masalah terbesar, terhubung ke temuan nyata. | skills/rendering-audition/SKILL.md (fase Review); skills/rendering-audition/references/scorecard-template.md bagian "Bukti revisi" |
| "Device ini memang disengaja — sudah kubilang begitu." | Justifikasi tanpa mekanisme audible yang konkret bukan justifikasi — itu pembelaan. Temuan cliche-register hanya boleh dijawab dengan revisi ATAU justifikasi audible spesifik (mekanisme yang bisa didengar kenapa device dipertahankan); "ini disengaja" polos tidak diterima. | skills/jazz-composing/references/cliche-register.md; skills/rendering-audition/references/scorecard-template.md bagian cliche |
| "Responder-nya sudah main, jadi dialognya pasti kerasa." | Response yang menabrak call (overlap) atau responder yang mengisi SEMUA gap sama-sama membunuh dialog — bukan jumlah not yang menentukan, tapi apakah aturan emas non-overlap dipenuhi. | skills/jazz-composing/references/call-and-response.md aturan emas |
| "Bass-nya sudah chord-tone semua (1-3-5-7), pasti aman secara teori." | Chord-tone yang benar tapi naik lurus per beat dan identik dari bar ke bar bukan groove line — itu tangga arpeggio mekanis. Teori benar bukan syarat cukup; wajib ada variasi antar bar (≤2x pola identik berturut-turut). | skills/jazz-composing/references/bass-idiom-neo-soul.md §Variasi antar bar |
| "Bass main di tiap beat, kan section-nya ada drum/groove jadi harus rapat." | Bass 100% on-beat justru membunuh pocket — sinkopasi/push (`beat` desimal, mis. `4.5`) dan space/rest yang disengaja adalah bagian dari groove, bukan kekosongan yang perlu ditambal. | skills/jazz-composing/references/bass-idiom-neo-soul.md §Sinkopasi & push, §Space/rest; skills/jazz-composing/references/contract.md §Beat desimal & pickup |
| "Bass-nya sudah pakai `degree`, tidak perlu tulis `octave` — biar engine yang atur." | `degree` tidak membawa informasi oktaf; tanpa `octave` eksplisit register default engine bisa mendarat terlalu tinggi (kasus nyata: `upright-bass` jatuh di C4-Bb4, jauh di atas register bass yang wajar). `octave` eksplisit wajib di tiap not bass. | skills/jazz-composing/references/contract.md §Register & oktaf; skills/jazz-composing/references/bass-idiom-neo-soul.md §Register |
| "Ghost note + pickup bikin bass-nya lebih hidup, tambahkan saja di intro yang pelan." | `artic: ghost` (velocity x0.5) ditumpuk dengan pickup di section ber-`dynamics` sangat rendah bisa membuat atenuasi menumpuk sampai not itu tenggelam di bawah ambang berguna sampel — "dirasakan, bukan didengar" berubah jadi "tidak didengar sama sekali". Hemat kombinasi ini di section dynamics rendah. | skills/jazz-composing/references/contract.md §Artikulasi berlaku di semua role; skills/jazz-composing/references/bass-idiom-neo-soul.md §Ghost note |
